# Codex 项目级配置使用指南

本目录存放 Codex 项目级 `AGENTS.md` 母版。

项目级配置只记录当前项目特有、且 Codex 无法从代码、构建文件、工具配置和现有文档中可靠推断的信息。

## 推荐初始化顺序

新项目建议按照以下顺序初始化：

1. 创建 Git 仓库。
2. 使用项目脚手架生成基础代码。
3. 安装基础依赖并确认项目能够正常构建或启动。
4. 将本目录中的 `AGENTS.md` 复制到目标项目根目录。
5. 填写已经明确的项目约束，删除不适用的占位章节。
6. 在 Codex App 中打开目标项目，或从目标项目根目录启动 Codex CLI。
7. 执行 `/init`，让 Codex 根据当前代码库检查并补充项目配置。
8. 人工审查、修正并精简最终的 `AGENTS.md`。

推荐顺序可以概括为：

```text
创建项目脚手架
    ↓
确认项目可以运行
    ↓
复制 AGENTS.md 模板
    ↓
填写已知项目约束
    ↓
执行 /init
    ↓
人工审核和精简
```

不建议在空仓库中直接执行 `/init`，因为此时缺少构建文件、依赖配置、目录结构和现有代码，Codex 能识别的信息有限。

## 复制模板

在本仓库根目录执行：

```bash
cp "agents/codex/project/AGENTS.md" <project>/AGENTS.md
```

也可以手动复制到目标项目根目录：

```text
<project>/AGENTS.md
```

## 执行 `/init`

复制并初步填写模板后，可以通过以下任一方式初始化。

### Codex App

使用 Codex App 打开目标项目，在对话输入框中执行：

```text
/init
```

### Codex CLI

从目标项目根目录启动 Codex：

```bash
codex
```

然后执行：

```text
/init
```

`/init` 生成的内容只是初稿。完成后应人工检查，不要未经审核直接接受或覆盖现有模板。

可以继续向 Codex 提出以下要求：

```text
请基于当前代码库检查并完善根目录 AGENTS.md。

要求：

1. 保留现有模板的职责边界和有效规则。
2. 只补充无法从代码、构建文件和工具配置中可靠推断的重要信息。
3. 不重复用户级 AGENTS.md 中已经存在的通用规则。
4. 不复述完整目录树、依赖列表或构建配置。
5. 明确项目认可的安装、启动、测试、构建和检查命令。
6. 明确修改完成后的验证要求和完成标准。
7. 不确定的内容不要自行编造，应标记为待确认或直接省略。
8. 删除未填写或不适用的占位章节。
9. 保持文件简洁、明确、可执行。
10. 先展示建议修改内容，未经确认不要直接覆盖文件。
```

## 已有项目

对于已经具备完整代码、构建配置和测试配置的项目，可以直接：

1. 复制项目级 `AGENTS.md` 模板。
2. 填写已知的项目约束。
3. 使用 Codex App 打开项目，或从项目根目录启动 Codex CLI。
4. 执行 `/init`，或要求 Codex 检查并完善现有文件。
5. 审查、验证并精简最终内容。

不要为了填写模板而重复抄写项目中已经存在的信息。

## Codex 如何加载项目指令

Codex 会将用户级指令和项目级指令组合后使用。

项目范围内，Codex 通常从 Git 仓库根目录开始，沿当前工作目录逐层查找指令文件：

```text
<repository>/AGENTS.md
<repository>/<module>/AGENTS.md
<repository>/<module>/<submodule>/AGENTS.md
```

越靠近当前工作目录的规则越具体，发生冲突时应以更具体的规则为准。

在同一个目录中，Codex 按以下优先级选择一个指令文件：

```text
AGENTS.override.md
    ↓
AGENTS.md
    ↓
配置的其他备用文件名
```

因此：

- 根目录 `AGENTS.md` 用于整个仓库共享的规则。
- 子目录 `AGENTS.md` 用于特定模块的补充或更具体规则。
- `AGENTS.override.md` 会优先于同目录的 `AGENTS.md`，适合明确替换该目录规则的场景。
- 不要在没有覆盖需求时随意使用 `AGENTS.override.md`。
- 指令文件应尽量精简，避免把大量项目文档直接堆入上下文。

## 验证 Codex 是否正确加载

可以在仓库根目录执行：

```bash
codex --ask-for-approval never "Summarize the current instructions."
```

也可以在 Codex 对话中要求：

```text
请列出本次会话已加载的 AGENTS.md 或 AGENTS.override.md，
并按加载顺序概括各文件中的有效规则。
```

如果修改了 `AGENTS.md` 后发现当前会话仍使用旧内容，应重新启动 Codex 会话，让它重新构建指令链。

## 项目级配置应该包含什么

适合写入项目级 `AGENTS.md` 的内容包括：

- 项目用途和不容易从代码中判断的业务边界。
- 重要目录的职责及模块之间的依赖方向。
- 项目认可的安装、启动、测试、构建、Lint 和类型检查命令。
- 不同修改范围对应的验证要求。
- 无法由 formatter、linter 或类型系统自动检查的代码约定。
- 不允许手工修改的生成代码或第三方目录。
- 修改源文件后必须同步生成的文件。
- 必须保持兼容的接口、数据结构或历史行为。
- 必须复用的公共组件、基础设施或架构模式。
- 非显而易见的业务规则。
- 常见陷阱和暂时不能修改的历史设计。
- 本地环境与 CI 行为之间的重要差异。

## 不应该包含什么

通常不需要写入项目级 `AGENTS.md` 的内容包括：

- 能从构建文件直接读取的完整依赖列表。
- 完整目录树或逐文件说明。
- formatter、linter 和类型系统已经强制执行的规则。
- README 或其他项目文档中已经清楚说明的普通介绍。
- 用户级 `AGENTS.md` 已经定义的沟通方式和通用 Git 工作流。
- 仅适用于某个开发者的编辑器、用户名和个人习惯。
- 本机绝对路径、密钥、令牌和其他敏感环境信息。
- 面向模板使用者的初始化教程。
- 模糊、无法验证或没有实际约束作用的口号式要求。

## 根目录与子目录规则

优先将整个仓库共享的规则放在根目录：

```text
<repository>/AGENTS.md
```

当某个模块确实存在独立规则时，再在对应目录增加：

```text
<repository>/services/payment/AGENTS.md
```

例如：

```text
<repository>/AGENTS.md
<repository>/frontend/AGENTS.md
<repository>/backend/AGENTS.md
```

根目录只保留公共约束，子目录只记录该模块特有的差异，避免在多个文件中复制相同规则。

只有在需要明确替换同目录常规规则时，才使用：

```text
<repository>/<module>/AGENTS.override.md
```

## 维护原则

- `AGENTS.md` 保持为可直接复制的纯模板。
- 本文件负责说明模板的使用方法，不要把使用教程写入母版。
- 模板负责统一结构，Codex 负责结合实际代码库提出具体内容，最终由人工审核决定。
- 项目经验只有在具备普遍复用价值时，才回写到母版。
- 某个项目独有的规则只保留在该项目中。
- 当 Codex 重复出现同一种错误时，再考虑把对应约束补入 `AGENTS.md`。
- 定期删除已经失效、已由工具强制执行或能从代码直接推断的规则。
- 修改模板结构后，同步检查本说明和仓库根目录 `README.md`。
