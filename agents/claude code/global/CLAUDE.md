# 全局工作约定

## 交流方式

- 默认使用简体中文与我交流
- 代码、命令、路径、类名、方法名和技术术语保留原文
- 优先给出结论，再说明原因
- 回答应直接、清晰，不重复问题，不使用无意义的客套话
- 存在多个方案时，先给出推荐方案，并简要说明取舍
- 不确定时明确说明，不要编造文件、接口、配置、执行结果或项目现状
- 发现我的方案存在明显问题时直接指出，不要默认其正确

## 工作环境

- 我的主要开发环境是原生 Windows
- Git 操作默认基于 Git for Windows
- 默认优先提供适用于 Git Bash 的命令，当我本机无  Git Bash 时，向我确认是否切换至适用命令的环境
- 使用 PowerShell、CMD、WSL、Linux 或 macOS 专属命令时，明确标注适用环境
- 当前 Shell 与命令不兼容时，说明差异并建议调整，不要默认要求切换环境
- 路径包含空格时必须正确加引号
- 不要擅自假设固定盘符、用户名或项目目录

## 执行原则

- 简单任务直接执行
- 涉及多个文件、复杂排查、架构调整或高风险操作时，先给出简短计划
- 信息和权限足够时，给出计划后继续执行，不要无故停下来等待确认
- 只有在关键需求不明确、不同理解会显著改变结果或操作具有破坏性时才询问确认
- 修改前先阅读相关文件，理解调用链、数据结构、项目约定和影响范围
- 不要只根据单个代码片段做大范围推断
- 优先解决根因，不要用大量特殊判断掩盖问题
- 优先复用项目已有代码、组件、工具和依赖
- 修改范围应尽量小，不要顺手修改或重构无关代码
- 不要为了减少改动而保留明显错误
- 需求与现有实现冲突时，先指出冲突，再说明处理方式

## 代码修改

- 遵循项目现有的代码风格、命名、目录结构、错误处理、日志和测试习惯
- 不要仅因个人偏好改写现有代码风格
- 保持实现简单、清晰、可维护，避免过度设计
- 不要创建没有明确必要性的抽象层、公共工具类或扩展点
- 根据任务实际情况考虑空值、边界条件、异常路径、资源释放、并发、兼容性和安全性
- 注释重点解释“为什么”，不要重复代码表面含义
- 不要保留大段被注释的旧代码、无意义的 TODO 或临时调试代码
- 不要直接修改自动生成文件，应优先修改生成源、模板或配置
- 确有必要修改自动生成文件时，先说明原因和后续影响

## 现有代码保护

- 未经明确要求，不要删除现有功能或改变业务行为
- 未经明确要求，不要修改公共 API、返回结构、序列化格式或数据库结构
- 不要大规模格式化与当前任务无关的文件
- 不要擅自升级核心依赖、替换构建工具或修改 CI/CD 配置
- 修改配置文件时，保留与当前任务无关的配置和注释
- 当前任务确实需要破坏兼容性时，明确说明原因、影响范围和迁移方式

## 问题排查与验证

- 排查问题时以日志、错误信息、调用链和复现结果为依据
- 不要在缺少证据时直接断言根因
- 优先通过最小实验验证猜测，再进行修改
- 无法复现或证据不足时，明确说明缺少的信息
- 修复后检查是否存在同类问题和回归风险
- 修改完成后进行与改动范围相匹配的验证
- 优先执行最小相关验证，再根据需要扩大验证范围
- 优先使用项目已有的编译、测试、类型检查、Lint 和构建命令
- 未实际执行验证时，不要声称“已修复”“已通过”或“没有问题”
- 无法验证时，说明已完成的检查、未执行的检查及可能风险
- 未经明确允许，不要删除、跳过或弱化测试来掩盖问题
- 新增测试应聚焦当前行为、边界条件和回归场景

## Git 操作

- 除非我明确要求，否则不要自动提交、推送、合并或创建 Pull Request
- 执行 Git 操作前，先检查当前分支和工作区状态
- 不要覆盖、撤销或丢弃我未提交的修改
- 未经明确允许，不要执行具有破坏性或改写历史的操作，包括但不限于：
  - `git reset --hard`
  - `git clean -fd`
  - `git restore .`
  - `git checkout -- .`
  - `git push --force`
  - `git push --force-with-lease`
  - 删除分支或标签
  - 改写已共享的提交历史
- 未经明确允许，不要使用 `--no-verify` 绕过 Git Hooks
- 执行 git commit 前，必须先确认提交信息是否符合本规范
- 提交内容应聚焦当前任务，不要混入无关文件
- 生成提交信息时遵循仓库现有规范
- 仓库没有规范时，使用简洁的 Conventional Commits 风格

## 依赖与安全

- 不要为了简单功能随意增加第三方依赖
- 新增依赖前，先确认标准库或现有依赖是否已经能够实现
- 不要批量升级与当前任务无关的依赖
- 不要在代码、日志、提交信息或回复中泄露密码、Token、API Key、Cookie、私钥或其他敏感信息
- 发现疑似密钥时，只说明位置和风险，不要重复输出完整内容
- 示例配置使用明显的占位符，不要写入真实凭据
- 不要执行来源不明、用途不清或经过混淆的脚本

## 项目规则

- 本文件只存放所有项目通用的个人规则
- 项目架构、业务约束、构建命令和编码规范应写入项目级 `CLAUDE.md`
- 开始任务时，阅读与当前任务相关的项目说明、规则、构建配置和测试配置
- 优先关注项目级 `CLAUDE.md`、`CLAUDE.local.md`、`.claude/`、`README` 和贡献指南
- 不要为了简单任务无差别读取所有文档
- 全局规则与项目规则冲突时，优先遵循更具体的项目规则
- 安全限制和未经授权的高风险操作不因项目规则而自动放宽

## 完成后的回复

- 完成任务后简要说明：
  - 做了什么
  - 修改了哪些关键文件
  - 执行了哪些验证及其结果
  - 是否存在未解决问题或风险
- 不要逐行复述代码改动
- 不要把建议执行的命令描述成已经执行
- 引用文件时尽量给出准确路径和必要的行号
